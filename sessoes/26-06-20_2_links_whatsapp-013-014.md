## Investigação C2 — `requireAdmin` frágil se session undefined (2026-06-20)

**Achado procede — código frágil, mas não explorável na configuração atual.**

- `server.ts:30`: `if (session?.user.role !== "admin")` — se `session` for `undefined`, `session?.user` → `undefined`, `undefined.role` → TypeError (o `?.` só protege até `user`; `.role` é acesso regular em `undefined`).

- Risco real: defesa em profundidade. Se ordem dos middlewares mudar, ou `requireAdmin` for usado sem `requireAuth` → crash.

- `mesas` (`middleware/auth.ts:84`) faz `if (!req.user)` primeiro → seguro.
- **Fix:** `if (!session || session.user.role !== "admin")` — 1 caractere, defesa em profundidade.

### ✅ C2 corrigido (2026-06-20)

**Fix aplicado:** `server.ts:30` `session?.user.role` → `!session || session.user.role`. Defesa em profundidade.
**Débito cross-app:** `apps/site/server/server.ts:39` mesmo padrão frágil → registrado como `D-SITE-REQUIREADMIN` no `backlog.md`.

## Investigação C7 — Advisory lock via `pool.query()` quebrado (2026-06-20)

**Achado tecnicamente correto, impacto prático muito baixo.**

- `db/migrate.ts`: `pool.query("SELECT pg_advisory_lock(...)")` — conexão A. `pool.query("SELECT pg_advisory_unlock(...)")` — pode ser conexão B.
- Advisory locks do PG são session-level: lock na conexão A só libera na conexão A. Unlock na conexão B retorna `false`.
- Lock só é liberado de fato por `pool.end()`.
- Site (`apps/site/db/migrate.ts:17,44`) tem o MESMO padrão frágil.
- **Fix:** `pool.connect()` → conexão dedicada para lock+unlock na mesma sessão.
- Site: mesmo débito cross-app.

### ✅ C7 corrigido (2026-06-20)

**Fix aplicado:** `pool.query()` → `pool.connect()` (lockClient). Lock+unlock na mesma sessão PG via nested try/finally.
**Débito cross-app:** `apps/site/db/migrate.ts` mesmo padrão → registrado como `D-SITE-ADVISORY-LOCK` no backlog.

## Investigação I2 — Shell `|| echo` quebra `set -e` (2026-06-20)

**Achado procede — `|| echo` mascara erros reais de seed. Mas o problema real é seed não-idempotente.**

- POSIX: `cmd1 || cmd2` — `set -e` é ignorado para `cmd1`. `echo` sempre sucede (exit 0) → script continua.
- Se DB estiver offline → seed falha → `|| echo` engole → rebuild tenta rodar → falha depois (pior, sem log claro).
- O `|| echo` existe como curativo: seed não é idempotente (INSERT de grupos e tags sem ON CONFLICT). Se dois deploys concorrentes → UNIQUE violation → `|| echo` salva.
- Site (`docker-entrypoint.sh`) NÃO usa `||` — confia em idempotência e deixa `set -e` matar.
- **Fix real:** (1) C8 → ON CONFLICT nas tags, (2) ON CONFLICT no INSERT de grupos do seed, (3) remover `|| echo`.

### ✅ I2 + C8 corrigidos (2026-06-20)

**Fix aplicado (3 partes):**
1. **`db/seed.ts:34`** — INSERT de tags + `.onConflict((oc) => oc.column("slug").doNothing())` (C8 resolvido).
2. **`db/seed.ts:76-92`** — INSERT de grupos + `.onConflict((oc) => oc.column("invite_url").doNothing())`.
3. **`docker-entrypoint.sh:18`** — removido `|| echo "... seed ignorado..."`. Seed verdadeiramente idempotente; `set -e` mata em erro real.

**Validação:** `tsc --noEmit` verde, `astro build` 15 páginas verde.
